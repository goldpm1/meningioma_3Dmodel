# Meningioma 3D Tumor Mapper - 작업 현황

## 목적
Meningioma 환자 24명(P01-P24)의 종양 위치를 뇌/척추 3D 모델 위에 클릭으로 표시하는 웹 도구.
기존 Illustrator 2D 수작업 그림을 대체. Nature/Science급 논문 그림 제작이 최종 목표.

## 현재 산출물 (중요: 파일 두 개가 있음, 헷갈리지 말 것)
- `meningioma_3d_tumor_mapper_pre_voxel.html` ← **지금부터 이게 메인 작업 파일임.**
  gray/white matter voxel 채우기 시도를 사용자가 포기하기로 결정해서, voxel 작업 이전
  (OneDrive 버전 기록에서 복원한) 상태로 되돌아왔고, 이후 모든 새 작업은 이 파일에 하고 있음.
- `meningioma_3d_tumor_mapper.html` ← voxel 방식으로 z-fighting까지 고쳤던 버전인데,
  사용자가 voxel 자체를 포기해서 더 이상 사용하지 않음. 참고용으로만 남겨둠, 새 작업 대상 아님.

두 파일 다 outputs 폴더 방식이 아니라 OneDrive 연결 폴더에 있음. 단일 파일, 더블클릭으로 실행,
서버 불필요. Three.js r128 기반.

## 완료된 작업 (2026-07-03, 최신)
`_pre_voxel.html`에 종양(tumor) 편집 기능을 추가함.

1. **회전**: 종양을 선택하면 사이드바에 X/Y/Z 회전 슬라이더(0~360도)가 뜸. 3D 화면에서는 선택된
   종양 옆에 파란 ↻ 핸들이 뜨는데, 이걸 드래그하면 카메라 기준 좌우/상하로 종양이 회전함(카메라를
   돌려가며 여러 번 드래그하면 어떤 방향이든 맞출 수 있음). 슬라이더와 드래그 둘 다 같은
   quaternion을 조작하므로 서로 값이 동기화됨. 회전값은 localStorage 자동저장/JSON
   내보내기·불러오기에 모두 포함되도록 함(원래 JSON 내보내기에는 회전이 빠져있었어서 추가함).
2. **삭제**: 선택된 종양 옆에 빨간 ✕ 핸들이 떠서 바로 클릭하면 삭제됨. 기존 사이드바
   "선택 항목 삭제" 버튼도 그대로 있음.
3. **단면(clip plane) 활성 상태에서만, Dura·Falx·Tentorium·Ventricle 표면에만 종양 부착
   가능**: 처음에는 "절단면 위 아무데나"로 제한했다가, 사용자가 "뇌수막종은 실제로 Dura, Falx,
   Tentorium, Ventricle에서 생기니까 그 위에만 붙어야 한다"고 요청해서 다시 고침. 지금은:
   - "단면으로 내부 보기"에서 시상/관상/축상 중 하나를 선택해야만 종양 추가 모드가 켜짐
     (꺼져있으면 체크박스 비활성화 + 주황 안내문).
   - 클릭한 마우스 광선을 현재 화면에 보이는 모든 구조물에 대해 raycasting해서 카메라에
     가장 가까운(=화면에서 실제로 보이는) 지점을 찾고, 절단면으로 잘려서 안 보이는 쪽에 있는
     교차점은 제외함(clipPlane과의 signed distance로 판정). 그렇게 찾은 "실제로 보이는 가장
     가까운 표면"이 Dura/Falx/Tentorium/Ventricles 넷 중 하나일 때만 종양을 붙임 — 즉 대뇌/
     소뇌/뇌간/척추/두개골에 가려져 있으면 그 뒤의 막 구조를 클릭해도 안 붙고, 대뇌 표면 등을
     직접 클릭해도 안 붙음. `raycastForAttachment()` 함수.
   - 종양의 방향은 부착면의 실제 face normal을 이용해 매번 정렬함(로컬 -Y가 표면 안쪽을
     향하도록) — fuji/en-plaque처럼 밑면이 평평한 모양은 이걸로 표면에 거의 딱 붙어 보이고,
     nodular/lobulated 같은 둥근 모양도 일관되게 "부착된" 방향을 가짐. 기존에 있던 point-in-mesh
     기반 구조 판별 코드(`structureAt`, `isValidTumorPoint` 등)는 이제 안 쓰여서 지웠음 — 대신
     맞은 mesh의 `userData.structure`를 그대로 씀(더 정확하고 간단함).
   - **주의(스코프 밖으로 남긴 부분)**: 사용자가 "tumor의 base가 underlying dura에 딱 맞게
     변형되어 부착"되길 원했는데, 이건 절반만 구현함. 표면 normal 방향 정렬은 했지만, tumor
     밑면 정점들을 실제 dura의 국소 곡률에 맞춰 개별적으로 구부리는(shrink-wrap) 것까지는 안 함
     — 그건 훨씬 큰 작업이라(mesh 정점별로 표면까지 거리 재서 변형) 이번에는 안 건드림. 지금은
     "평평/오목한 밑면 + 표면 normal 정렬"로 근사한 상태. 사용자가 이 근사로 부족하다고 하면
     추가로 구현 필요.

검증한 것: JS 문법 파싱 통과, 새로 추가/변경한 DOM id·함수 참조가 다 존재하는지 확인, 죽은 코드
정리.

검증하지 못한 것(사용자가 직접 열어서 확인 필요): 실제 브라우저에서 Dura/Falx/Tentorium/Ventricle
위 클릭이 잘 되는지, 다른 구조에 가려졌을 때 정말 거부되는지, 회전 드래그 감도, en-plaque/fuji
모양이 표면에 자연스럽게 붙어 보이는지.

## 완료된 작업 (2026-07-03, 최신 2 - 크기 조절 + 레이아웃 재정리)
1. **화면에서 바로 크기 조절**: 선택된 종양 옆에 초록색 ⤢ 핸들이 추가됨. 드래그(위=커짐/
   아래=작아짐)하면 `직경`이 바로 바뀜(geometry 재생성, `regenerateTumorGeometry()`).
2. **X/Y/Z 축별 크기 배율 슬라이더**(사이드바, "선택된 종양 편집" 안): 0.3~3.0배, 종양 자체의
   로컬 X/Y/Z를 독립적으로 늘리고 줄일 수 있음. geometry를 다시 만드는 게 아니라 `mesh.scale`을
   써서 가벼움. 저장/불러오기(localStorage, JSON) 다 반영함. 회전이 적용된 상태에서는 이
   X/Y/Z가 "화면 기준"이 아니라 "종양 자체 기준(로컬)" 축이라는 점 참고.
3. **종양 목록을 사이드바에서 3D 뷰 오른쪽의 독립 패널로 분리**함(`#tumorListPanel`).
4. **레이아웃 정리**: "종양 추가 모드 (단축키 T)" → "Add Tumor (T)"로 이름 변경. 라벨 입력칸을
   "새 종양 설정" 섹션 제목 줄 오른쪽으로 이동(한 줄에 제목+라벨칸). "단면으로 내부 보기"를
   "구조 표시/숨기기" 바로 아래로, "Mesh 직접 편집"은 그 아래(단면 다음)로 순서 변경.

검증한 것: JS 문법 파싱, 새로 추가된 DOM id 전부 실제 HTML에 존재하는지 확인.
검증 못한 것: 실제 화면에서 크기 핸들 드래그 감도, X/Y/Z 슬라이더로 만든 비대칭 모양이 자연스러운지,
새 레이아웃이 보기 불편하지 않은지 — 사용자 확인 필요.

## 완료된 작업 (2026-07-03, 최신 3 - 축별 3D 핸들 + 재부착 + 모양 리얼리즘)
1. **화면에 X/Y/Z축 크기 핸들 3개 추가**(빨강/초록/파랑 원, 3D 상에서 서로 90도를 이루도록 종양의
   현재 회전을 반영해서 위치 계산함). 세로로 드래그하면 해당 축만 커지거나 작아짐 — 사이드바
   슬라이더와 완전히 같은 값(`scaleX/Y/Z`, `mesh.scale`)을 조작하므로 서로 동기화됨.
   `axisHandleWorldPos()`가 매 프레임 종양 위치 + (로컬축 회전 적용) + 반지름*배율 로 3D 좌표를
   구하고 화면에 투영함.
2. **크기 조절 후 Dura/Falx/Tentorium/Ventricle에 다시 딱 붙도록 재조절**: 종양을 배치하거나
   드래그로 옮길 때마다 그 순간의 부착점(`attachPoint`)과 표면 normal(`attachNormal`)을
   기억해두고, 지름 변경이든 축별 배율 변경이든 크기가 바뀔 때마다 `reattachToSurface()`를
   호출해서 "부착점 + normal 방향으로 (반지름×Y축배율×0.55)만큼 띄운 위치"로 다시 스냅함. 축
   핸들 드래그, 사이드바 크기/배율 슬라이더, 크기 초기화 버튼, 지름 리사이즈 핸들, "선택 항목
   업데이트" 버튼에 전부 연결함. localStorage/JSON으로 불러온 옛날 종양처럼 attachPoint가 없는
   경우는 조용히 건너뜀(기존 위치 유지).
3. **종양 모양을 실물 육안 소견 사진처럼 개선**(외부 3D 에셋/텍스처는 필요 없었음, 순수 procedural
   로 해결): 기존에는 단순 sine 노이즈로 만든 매끈한 울퉁불퉁한 덩어리였는데, 이번에 Fibonacci
   sphere로 만든 "소엽(lobule) 중심점" 여러 개를 종양마다 무작위 회전시켜 배치하고, 각 정점에서
   가장 가까운 소엽 중심까지의 각도 기반 영향력(Worley/cellular noise 비슷한 방식)을 계산해서
   방울방울 뭉친 카울리플라워(콜리플라워) 모양의 lobule을 만듦. nodular(기본)는 소엽 6개로
   은은하게, lobulated는 11~15개로 확실하게 울퉁불퉁하게, fuji/en-plaque도 각각 어울리는 정도로
   적용함. 종양마다 소엽 배치와 미세 텍스처 위상(phase)이 매번 랜덤이라 똑같이 생긴 종양이
   반복되지 않음(단, 이 랜덤성은 저장 안 되므로 새로고침하면 같은 종양이라도 모양이 살짝 달라질
   수 있음 — 문제라고 하면 seed를 저장하도록 고칠 수 있음). Vertex color 같은 재질 쪽 변경은
   안 했고(three.js 버전별 API 차이 리스크 있어서), 새로 생긴 실제 굴곡(geometry)이 기존 조명
   만으로도 소엽 사이 그늘을 자연스럽게 만들어줄 것으로 예상함.

검증한 것: JS 문법 파싱, 새 DOM id 확인, displacement 범위 수치 계산으로 지오메트리가 뒤집히거나
0/음수 반지름이 되지 않는지 확인.
검증 못한 것: 실제 화면에서 축 핸들 3개의 위치/드래그가 기대대로 되는지, 재부착이 자연스러운지,
새 종양 모양이 참고 사진과 비슷한 느낌인지 — 사용자 확인 필요.

## 버그 수정 (2026-07-03, 핸들이 전혀 안 보이던 문제)
사용자가 실제로 열어보니 종양 모양은 바뀌었는데 핸들(삭제/회전/크기/축별)이 하나도 안 보인다고
보고함. 이 세션엔 브라우저가 없어서 직접 화면을 볼 수 없었음 — 대신 사용자에게 콘솔에 진단 코드를
붙여넣게 해서 `tumorHandles`/`axisHandles`의 class와 style을 확인함. 결과: 종양은 분명히 선택된
상태(목록에 active 1개)인데 두 요소의 class가 빈 문자열이고 style도 null.

원인: 이 파일에 원래부터 있던 올가미(lasso) 삭제 기능이 `function worldToScreenPx(x,y,z)`라는
함수를 쓰고 있었는데, 핸들 기능을 추가하면서 완전히 다른 용도의 `function worldToScreenPx(worldPos,
rect)`를 같은 이름으로 새로 만들어버림. 자바스크립트에서 같은 스코프의 `function` 선언은 나중 선언이
앞의 것을 덮어써버려서(변수 재할당과 같은 원리), 파일 뒤쪽에 있던 올가미용 버전이 핸들용 버전을
조용히 덮어씀 — 에러 없이 그냥 잘못된 함수가 호출됨. 그래서 핸들 위치 계산이 전부 이상한 값(NaN)이
됐고, 안전장치로 넣어둔 isFinite 체크가 매 프레임 핸들을 숨겨버렸음(class가 비어있던 이유).

수정: 새로 만든 함수 이름을 `tumorWorldToScreenPx`로 바꿔서 충돌 제거함. build 태그를
`2026-07-03-axis-handles-v2-namefix`로 올림 — 사이드바 제목 밑, 콘솔 로그 둘 다에 표시되니 사용자가
최신 버전인지 바로 확인 가능.

교훈: 이렇게 큰 단일 파일에 계속 함수를 추가하다 보면 이름 충돌이 생기기 쉬움. 다음에 새 함수를
추가할 때는 되도록 접두어(예: `tumorXxx`, `lassoXxx`)를 붙이거나, 추가하기 전에 grep으로 같은 이름이
이미 있는지 확인하는 습관이 필요함.

## 완료된 작업 (2026-07-03, 최신 4 - 선택 해제 + 저장 상태 표시)
1. **종양 이외의 부분을 클릭하면 새 종양이 생기지 않고 선택 해제만 되도록**: 기존에는 종양을
   선택/수정한 뒤 다른 곳을 클릭하면 (추가 모드가 켜져있으면) 새 종양이 바로 생겨버렸음. 이제는
   뭔가 선택된 상태에서 다른 곳을 클릭하면 무조건 선택 해제(중립 상태)만 되고, 새 종양은 안 생김.
   아무것도 선택 안 된 상태에서 유효한 표면을 클릭해야만 새로 추가됨 — 즉 "다른 데 클릭 → 해제"와
   "새로 추가"가 분리됨(먼저 클릭으로 해제하고, 한 번 더 클릭해야 추가).
2. **저장 상태를 눈으로 확인할 수 있게 함**: 원래도 종양을 추가/이동/수정할 때마다 자동으로
   `localStorage`에 저장되고 있었고(브라우저에 남아있음), 페이지를 다시 열면 `loadLocal()`이 자동
   복원하고 있었음 — 이 자체는 이전부터 있던 기능인데 사용자가 눈으로 확인할 방법이 없었음. 이제
   오른쪽 "종양 목록" 패널 아래에 작은 문구가 생겨서, 저장될 때마다 "저장됨 (n개)"가 잠깐 떴다가
   기본 문구("브라우저에 자동 저장됨...")로 돌아가고, 페이지를 열 때 이전 데이터를 불러오면
   "이전에 저장된 종양 n개를 불러왔습니다"가 뜸. 만약 브라우저가 localStorage를 막고 있으면(예:

삼아서, **원본**(제가 손대기 전) Spine 데이터를 기준으로 오프셋을 완전히 새로 계산함(이전 2번의
잘못된 이동을 누적하지 않고, 처음부터 다시 계산). 새 오프셋 ≈ x+5.3, y-38.4, z+13.9mm(원본 기준).
결과적으로 지난 수정 대비 Spine이 뒤로(−Z 방향) 약 64mm, 아래로 약 11mm 더 이동한 셈 — 사용자가
말한 "뒤로 많이, 아래로도"와 방향이 일치함. Skull은 지난번 삭제 상태 그대로 안 건드림.
`meningioma_3d_tumor_mapper_pre_voxel.html`에 반영.

검증한 것: JS 문법, vertex/face 인덱스, y/z 범위(Skull -94.99~70.25 / Spine y -753.07~-88.27,
z -176.44~-15.01 — 이제 Spine 전체가 z<0, 즉 후방에 위치함 확인). Skull과 Spine 겹치는 부분도

검증 못한 것: 실제 화면에서 잘 붙어 보이는지 — 스크린샷 필요. 회전(뒤틀림)은 여전히 안 건드림.

## 남은 미해결 이슈
- Foramen magnum/C1 부근에서 두개골이 복측(ventral) dura를 뚫고 나오는 문제 (자동 수정 시도가
  심하게 왜곡돼서 되돌려 놓은 상태, 정확한 위치 스크린샷 필요) — 위 Spine/Skull 정렬 문제와 같은
  부위라 스크린샷 하나로 두 문제를 같이 살펴볼 수 있을 듯
- Spine mesh 위쪽(C1/C2)의 뒤틀린/울퉁불퉁한 모양 자체는 이번에도 안 고쳤음(위치만 옮김) — 필요하면
  나중에 다시 볼 것
- Skull과 Spine이 세로로 약 6.7mm 겹치는 부분이 시각적으로 괜찮은지 확인 필요
- gray matter fill(GrayFill_L/R)은 `_pre_voxel.html`에는 아예 없음(빈 문자열) — voxel을
  포기했으니 이 부분을 다시 어떻게 할지는 아직 정해지지 않음
- tumor 밑면을 dura 국소 곡률에 맞춰 실제로 변형(shrink-wrap)하는 것 — 위 "스코프 밖" 참고
- 위 "검증하지 못한 것" 항목들, 사용자 브라우저 확인 필요
